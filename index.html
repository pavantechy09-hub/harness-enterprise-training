🤖What is the Harness Delegate?
The Problem It Solves
Harness SaaS lives at app.harness.io. Your cluster lives at localhost. The internet cannot reach localhost. The Delegate solves this by running inside your cluster and polling Harness outbound. All traffic is outbound — no inbound firewall rules needed.
polling loop (every 10s)
Delegate → GET https://app.harness.io/tasks?delegateId=dd-stg
Harness  → "Run: terraform apply in day7-security/"
Delegate → executes locally (has access to localhost:4566)
Delegate → POST results back to Harness
Harness  → shows step passed/failed in UI
Enterprise Pattern — Multiple Delegates
Production: 2 Delegates minimum (HA), tagged env=prod, has prod credentials only.
Staging: 1 Delegate, tagged env=staging.
Dev/local: dd-stg-delegate on your laptop, tagged env=dev.

Pipeline steps use selector: env=prod → only prod Delegates run prod steps. Security enforced at infrastructure level.

❓What is a Harness Delegate and why is it needed?▼
A Delegate is an agent pod running inside your Kubernetes cluster that polls Harness SaaS for tasks and executes them locally. It is needed because Harness SaaS lives on the internet but your infrastructure — clusters, Floci, internal APIs — lives behind firewalls. The Delegate makes all traffic outbound only, so no inbound firewall rules are needed, your credentials stay inside your network, and it works in air-gapped environments. Enterprise teams run 2+ Delegates per cluster for high availability.
❓What is the Harness entity hierarchy?▼
Account is the top level. Under it you create Organisations (e.g. firstnational-bank) representing business units. Under each Org you create Projects (e.g. bank-platform) which are collections of related pipelines and resources. Inside a Project you define Services (what to deploy), Environments (where to deploy), Infrastructure Definitions (how to connect), and Pipelines (orchestration). This hierarchy enables fine-grained RBAC — a developer might access one Project but not another.
❓How do Delegate selectors work in enterprise?▼
Delegates are tagged with key-value pairs like environment=prod or cluster=eks-us-east-1. Pipeline steps specify which Delegates they should run on using these selectors. A prod deployment step with selector environment=prod will only execute on Delegates tagged with that value. This means prod credentials never touch dev Delegates, providing security isolation at the infrastructure level without any application-level code changes.

❓What is Harness Test Intelligence?▼
Test Intelligence is an AI feature that analyzes your code changes and identifies which tests are actually relevant to those changes. Instead of running all 10,000 tests on every commit, it might run only the 300 tests affected by your change. This typically reduces CI time by 60-80%. It uses code graph analysis to understand which classes and functions your change touches, then maps those to test files that import or call those components.
❓How does Harness STO differ from just running Checkov in a shell step?▼
Harness STO (Security Testing Orchestration) provides a structured, normalized view of security findings across multiple scanners. When you run Checkov as a raw shell step, the output is just text in logs. STO parses the results, categorizes them by severity, de-duplicates findings across scans, tracks them over time, and shows them in a dedicated security dashboard. It can block pipelines on HIGH severity findings with a single fail_on_severity: HIGH config rather than parsing exit codes manually.

❓What are the main risks when migrating from Jenkins to Harness?▼
The highest technical risk is leaving teams without a deployment path during migration. Mitigate by running Jenkins and Harness in parallel for at least 2 weeks per service. The highest organisational risk is developer resistance — teams comfortable with Jenkinsfiles may resist change. Mitigate with early involvement, training, and concrete demonstrations of benefits like automatic rollback and faster deployments. A genuine technical constraint is that Harness Delegates are Linux-based, so Windows build agents require containers or staying on Jenkins during initial migration phases.
❓How do you handle Jenkins shared libraries when migrating?▼
Jenkins shared libraries are reusable Groovy code that teams import into Jenkinsfiles. The Harness equivalent is Pipeline Templates. You define a template once — for example a standard CI template for all microservices — and each pipeline references it with a version tag. When you update the template, all pipelines using it pick up the change on next run. This is actually better than Jenkins shared libraries because templates are versioned, have their own approval process, and are visible in the UI rather than hidden in a separate repository.

❓Explain canary deployment end-to-end in a banking context.▼
A canary deployment gradually shifts traffic from old to new version. For payments-service we deploy v1.1.0 alongside v1.0.0. Harness routes 10% of transactions to the new version, monitoring error rates and p99 latency. If metrics stay healthy, Harness promotes to 25%, 50%, then 100%. If checks fail, automatic rollback fires and only 10% of users were exposed. This is why banks use canary for payment changes — even if broken, blast radius is a fraction of customers, not everyone.
❓When would canary NOT be the right strategy?▼
Canary is wrong when mixed versions cause data corruption — e.g. a DB schema migration where new code writes a field old code doesn't understand. Also wrong for breaking API contract changes. In these cases blue-green is safer — zero real users see new version until full validation and switch. Feature flags are better when the change is behavioural and can be toggled per user without redeployment.

❓Explain blue-green and when to use it over canary.▼
Blue-green maintains two complete identical environments. Blue is current production at 100% traffic. Green is deployed with new version at zero traffic. After full smoke testing of Green, traffic switches instantly via Kubernetes Service selector update completing in under 3 seconds. Blue stays alive 30 minutes for rollback. Use over canary when instant rollback is required for compliance — our accounts-service changes must be reversible within 60 seconds per FCA regulation. Also use when mixed versions cause data inconsistency.
❓How does Harness implement blue-green in Kubernetes technically?▼
Harness creates two Deployments labelled harness.io/color: blue and harness.io/color: green. Two Services: primary pointing to live colour, stage pointing to non-live for smoke testing. Traffic switch = kubectl patch service accounts-service updating selector from blue to green. Propagates to kube-proxy in 1-3 seconds with zero dropped requests. Rollback = same operation in reverse.
❓What is the main cost disadvantage of blue-green?▼
Requires double compute during deployment — both Blue and Green run full replica counts simultaneously. accounts-service with 3 replicas means 6 pods during switch window. This is why blue-green is reserved for compliance-critical services where safety justifies cost, while stateless services use rolling deployments with zero overhead.

❓What is Harness and how does it differ from GitHub Actions?▼
Harness is an enterprise CD platform built specifically for production-grade delivery. GitHub Actions is excellent for CI but hits limits for enterprise CD. Harness adds built-in canary and blue-green strategies with automatic rollback, OPA governance on every pipeline, SLO tracking with error budgets, cloud cost management, and AI assistance (AIDA). The key difference: Harness treats the deployment itself as a first-class concern — it knows what healthy looks like, monitors it, and rolls back automatically if health checks fail.
❓When would you use canary vs blue-green vs rolling?▼
Canary is best for high-risk changes to critical services like payments — gradually expose a small percentage of real users, monitor closely, commit when confident. Blue-green is best when you need instant rollback capability — deploy a complete new environment and switch traffic instantly, keep old environment alive for quick reversion. Rolling is the default for stateless services where brief mixed-version periods are tolerable. In our bank platform: canary for payments-service (financial risk), blue-green for accounts-service (compliance requires instant rollback), rolling for fraud and notification services.
❓What is an OPA policy in Harness and how does it work?▼
OPA (Open Policy Agent) is a policy-as-code engine using the Rego language. Harness evaluates every pipeline step against configured OPA policies before execution. We used it to enforce: production requiring two approvers from different teams, no self-approval of your own code, Checkov HIGH findings blocking any deployment, no production deployments outside business hours, and mandatory resource limits on all K8s manifests. All policies are stored in Git giving a complete audit trail of every governance decision — which is exactly what compliance teams require for SOX, PCI-DSS, and other regulations.
❓What is the difference between SLI, SLO, and SLA?▼
An SLI (Service Level Indicator) is the actual metric measured — for example the fraction of HTTP requests returning 200 status. An SLO (Service Level Objective) is the internal target — 99.9% of requests must succeed. An SLA (Service Level Agreement) is the external contract with customers carrying financial penalties if breached. In Harness SRM we configure SLIs and SLOs, which track error budgets (the amount of failure you can afford). When 90% of the budget is consumed, deployments are automatically blocked before any SLA violation can occur.
❓How would you approach migrating from Jenkins to Harness in 90 days?▼
Three phases. Phase 1 (Days 1-30): Install Delegates in non-prod, migrate one non-critical service as POC, run Jenkins and Harness in parallel, train all teams. Phase 2 (Days 31-60): New services go Harness-only from day one, migrate existing services in batches starting with lowest risk, keep each Jenkins pipeline as fallback for 2 weeks before removing. Phase 3 (Days 61-90): Migrate critical services like payments with extended parallel running and validated rollback testing, then decommission Jenkins. Key principle: never leave teams without a rollback option during the transition window.
❓What are DORA metrics and how does Harness improve them?▼
DORA metrics are the four industry-standard measures of DevOps performance. Deployment Frequency — how often you deploy. Lead Time for Changes — code commit to production. MTTR — time to recover from incidents. Change Failure Rate — percentage of deployments causing incidents. Harness improves all four: automated pipelines increase deployment frequency, removing manual steps cuts lead time, automatic rollback drastically reduces MTTR, and canary deployments catch issues before full rollout reducing change failure rate. Measuring these before and after migration provides concrete ROI justification for leadership.
