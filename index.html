🤖Delegate — The Most Important Concept
The Problem Delegate Solves
Harness SaaS lives at app.harness.io on the internet. Your Kubernetes cluster lives at localhost on your laptop (or in a private VPC in production). The internet cannot reach localhost or a private VPC. The Delegate bridges this gap.
Without Delegate vs With Delegate
Harness SaaS
app.harness.io
✗
Your Cluster
localhost (blocked)
With Delegate — all traffic is OUTBOUND from your side:
Delegate Pod
inside your cluster
→ polls every 10s
Harness SaaS
gets tasks
→ executes locally
Your Cluster
full access ✓
🔒
Security Benefit
No inbound firewall rules needed. No VPN. No public IP on your cluster. All traffic is outbound on port 443. Security team happy. Works behind corporate firewalls.
🔄
Polling Loop
Delegate calls Harness every 10 seconds: "Any tasks for me?" Same concept as Kubernetes controller reconciliation loop. If Delegate goes offline — tasks queue in Harness until it reconnects.
🏷️
Selector Tags
Delegates have tags like env=prod or env=dev. Pipeline steps target specific tags. Prod pipelines ONLY run on prod Delegates. Dev credentials never touch prod.
⚖️
vs GitHub Actions Runner
Same concept as self-hosted GitHub Actions runner. But Delegate: auto-upgrades itself, auto-selects based on tags, understands native Harness step types, load balances across multiple instances.

🔌The 5 Connectors — What Each Does
Mental model: Delegate = the executor (runs tasks). Connectors = the credentials + endpoints (tells Delegate WHERE and HOW to connect). Together they form the complete execution engine.
🤖
Harness Delegate (helm-delegate)
TYPE: Agent Pod · NAMESPACE: harness-delegate-ng
Not technically a connector but the foundation of everything. Runs inside dd-stg cluster. Polls Harness every 10s for tasks. Executes all steps locally using the other connectors for credentials.
🐙
GitHub Connector (bankopsgithub)
ID: bankopsgithub · URL: github.com/pavantechy09-hub/harness-enterprise-training
What it stores: Your GitHub Personal Access Token (encrypted in Harness Secrets Manager).
What it does: Delegate uses this to clone your repo into each build pod during "Clone codebase" step.
Without it: Harness cannot read your application code. Every build fails at clone step.
☸️
Kubernetes Connector (ddstgk8s)
ID: ddstgk8s · CLUSTER: kind-dd-stg
What it stores: Permission to talk to Kubernetes API (uses Delegate service account).
What it does: Delegate uses this to CREATE build pods in harness-builds namespace for each pipeline run. Pods are deleted after build completes.
Without it: Harness cannot create build environment. All pipeline steps fail with "empty connector" error.
🐳
Docker Hub Connector (dockerhub)
ID: dockerhub · URL: index.docker.io/v1/
What it stores: Docker Hub credentials (anonymous for public images).
What it does: Build pods use this to pull python:3.12-slim image before running your test command.
Without it: Run steps fail with "Run step with Kubernetes infra can't have empty connector field".
🔐
Harness Built-in Secrets Manager
ID: harnessSecretManager · AUTO-CREATED
What it does: Encrypts all credentials stored in Harness. GitHub token, Docker password — never stored in plain text. Reference in pipelines as <+secrets.getValue("my-secret")>.
Enterprise upgrade: Replace with HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault.

2. Shared Pipeline Templates — Like GitLab Shared Templates
Without templates: 4 services = 4 duplicate CI configs. If you fix a bug in test logic, you fix it 4 times. With templates: define once, all services inherit. One change propagates everywhere automatically.
.harness/templates/standard-ci-template.yaml
# Defined ONCE — used by ALL 4 services
template:
  name: Standard CI Template
  identifier: standard_ci_template
  versionLabel: v1.0
  type: Stage
  spec:
    type: CI
    spec:
      execution:
        steps:
          - step:
              name: Run Tests
              type: Run
              spec:
                connectorRef: dockerhub
                image: python:3.12-slim
                command: |
                  cd src/<+stage.variables.service_name>
                  pip install -r requirements.txt -q
                  pytest app/tests/ -v

      variables:
        - name: service_name    # overridden per service
          type: String
          value: <+input>

5. OPA Governance Policies
.harness/policies/prod-deployment-policy.rego
# Block prod without 2 approvals (four-eyes principle)
deny[msg] {
  input.pipeline.stages[_].stage.spec.environment.identifier == "prod"
  count(input.approvals) < 2
  msg := "Production requires 2 approvals — four-eyes principle"
}

# Block self-approval
deny[msg] {
  input.pipeline.triggeredBy == input.approvals[_].approvedBy
  msg := "Cannot approve your own deployment"
}

# Block outside business hours
deny[msg] {
  input.pipeline.stages[_].stage.spec.environment.identifier == "prod"
  hour := time.clock(time.now_ns())[0]
  hour >= 17
  msg := "No production deployments after 5pm"
}

# Block if Checkov has HIGH findings
deny[msg] {
  input.securityScan.status == "FAILED"
  input.securityScan.severity == "HIGH"
  msg := sprintf("HIGH security finding blocks deployment: %s", [input.securityScan.checkId])
}
🐛Issues We Hit — Root Cause + Fix + Lesson
🔴 Issue 1: Run step with Kubernetes infra can't have empty connector field
What happened: Used connectorRef: account.harnessci — a default connector name that doesn't exist in our account.

Root cause: Every connector has a unique ID generated when you CREATE it. You cannot guess or assume the ID. The Run step requires a valid Docker registry connector to know where to pull container images from.

Fix: Created Docker Hub connector manually (Project Settings → Connectors → Docker Registry). Got actual ID: dockerhub. Updated YAML.

Lesson: Always create ALL connectors BEFORE writing pipeline YAML. Then reference their actual IDs. In enterprise, connectors are created once by platform team and shared across projects.
🔴 Issue 2: namespaces "harness-builds" not found
What happened: Harness tried to create a build pod in harness-builds namespace. Pipeline failed immediately at Initialize step.

Root cause: Harness does NOT auto-create Kubernetes namespaces. This is a Kubernetes security principle — namespaces are isolation boundaries and should be created deliberately, not by automation.

Fix: kubectl create namespace harness-builds then re-run pipeline.

Lesson: In enterprise, platform team pre-creates all namespaces with proper RBAC before pipelines run. Add namespace creation to your cluster bootstrap scripts, not your pipelines.
🟡 Issue 3: PowerShell backtick multiline helm command failed
What happened: Helm install command using PowerShell backtick (`) for line continuation appeared to run but created no pods.

Root cause: PowerShell backtick line continuation is extremely sensitive to trailing whitespace after the backtick. A single space after ` breaks the continuation silently.

Fix: Run entire helm command as single line without any line breaks.

Lesson: For complex PowerShell commands, use single line or use a .ps1 script file. Never rely on backtick continuation for critical commands.
🟡 Issue 4: Helm not found after choco install
What happened: Chocolatey installed helm but helm version failed with "not recognized".

Root cause: Chocolatey installs to a non-standard path. PATH environment variable not refreshed in current PowerShell session after install.

Fix: Manually found helm.exe location, added to PATH:
$env:PATH += ";C:\Program Files\...\kubernetes-helm ools\windows-amd64"

Lesson: After choco install, either open new PowerShell window or manually refresh PATH. On Linux/Mac this is not an issue — brew/apt update PATH automatically.

❓Interview Questions — Day 1 Concepts
❓What is a Harness Delegate and why is it needed?▼
A Delegate is an agent pod running inside your Kubernetes cluster that polls Harness SaaS for tasks and executes them locally. It exists because Harness is a SaaS platform on the internet but your infrastructure — clusters, private registries, internal APIs — lives behind firewalls. The Delegate makes all communication outbound only from your side, so no inbound firewall rules are needed, no VPN is required, and your cluster needs no public IP. In enterprise, you run 2+ Delegates per cluster with different selector tags so prod pipelines only run on prod Delegates, completely isolating environments at the infrastructure level.
❓What is the difference between a Delegate and a Connector in Harness?▼
Delegate = the executor. It runs inside your network and has access to everything locally. It receives tasks from Harness and runs them. Connectors = credentials + endpoints. They tell the Delegate WHERE to connect and with WHAT credentials. For example: when running a build, the Delegate uses the GitHub connector to clone code, the Docker connector to pull images, and the Kubernetes connector to create build pods. The Delegate knows HOW to run commands — the connectors know WHERE to connect and authenticate. You need both working together for any pipeline step to execute.
❓How does team isolation work in Harness for a large enterprise?▼
Harness uses a three-level hierarchy: Account → Organisation → Project. Each Project is a complete isolation boundary. The payments team only has access to the payments-platform project — they cannot see or run fraud-platform pipelines. Secrets are project-scoped, so payments credentials never touch fraud pipelines. RBAC roles control exactly what each team member can do within their project. Shared resources like Docker Hub connectors and standard CI templates live at Account level and are accessible by all projects. This mirrors how large banks operate: 50 teams, 50 projects, one account, with platform team managing shared resources centrally.
❓Should you store pipeline code in Harness or GitHub? Why?▼
Always GitHub (or GitLab/Bitbucket) for any production or team pipeline. Storing inline in Harness means pipeline changes have no audit trail — anyone with Harness access can silently change a deployment pipeline. GitHub storage means every pipeline change goes through a PR, gets reviewed by the right people via CODEOWNERS, has a full git history, and is protected by the same governance as application code. For compliance frameworks like SOX and PCI-DSS, you must prove who changed the deployment pipeline and when — git history provides this automatically. Inline storage is only appropriate for learning and short-lived experiments.
❓What is the CODEOWNERS file and how does it enforce governance?▼
CODEOWNERS is a GitHub file that automatically assigns required reviewers based on which files a PR changes. If a developer modifies src/payments-service/, GitHub automatically requests review from @payments-lead. The PR cannot be merged until that person approves. In enterprise this means: a developer cannot change the deployment pipeline without the platform team reviewing it, infrastructure changes require the security team, and application changes require the service lead. Combined with Harness approval gates on the CD side, this creates a complete chain of custody from code change to production deployment with every decision documented and attributed.
❓What are Harness Pipeline Templates and why do enterprises use them?▼
Templates are reusable pipeline building blocks defined once and referenced by multiple pipelines. Instead of 4 microservices each having their own copy of the same CI logic, you define one standard CI template with a variable for the service name. Each service pipeline references the template and passes its service name as input. When the platform team improves the template — adds security scanning, updates the Python version — all 4 services automatically get the improvement. This is identical to GitLab shared pipeline templates. It eliminates duplication, enforces standards across all services, and means a single change can propagate to 50 services simultaneously.
❓Walk me through what happens when a developer pushes code in your setup.▼
Developer pushes to feature branch. GitHub sends a webhook to Harness. Harness checks if any pipeline is triggered by this event — in Day 2 we add a PR trigger. Harness assigns the CI task to helm-delegate (our Delegate running on dd-stg cluster). Delegate creates a build pod in harness-builds namespace using the ddstgk8s connector. The pod clones the repo using the bankopsgithub connector. It pulls python:3.12-slim using the dockerhub connector. It runs pip install and pytest. Results — pass or fail — are reported back to Harness UI. The pod is deleted automatically. The whole flow is automated, audited, and repeatable. The developer sees results in the Harness UI or gets a Slack notification if it fails.

❓What is Harness Test Intelligence?▼
Test Intelligence is an AI feature that analyzes your code changes and identifies which tests are actually relevant to those changes. Instead of running all 10,000 tests on every commit, it might run only the 300 tests affected by your change. This typically reduces CI time by 60-80%. It uses code graph analysis to understand which classes and functions your change touches, then maps those to test files that import or call those components.
❓How does Harness STO differ from just running Checkov in a shell step?▼
Harness STO (Security Testing Orchestration) provides a structured, normalized view of security findings across multiple scanners. When you run Checkov as a raw shell step, the output is just text in logs. STO parses the results, categorizes them by severity, de-duplicates findings across scans, tracks them over time, and shows them in a dedicated security dashboard. It can block pipelines on HIGH severity findings with a single fail_on_severity: HIGH config rather than parsing exit codes manually.

❓What are the main risks when migrating from Jenkins to Harness?▼

❓How do you handle Jenkins shared libraries when migrating?▼
Jenkins shared libraries are reusable Groovy code that teams import into Jenkinsfiles. The Harness equivalent is Pipeline Templates. You define a template once — for example a standard CI template for all microservices — and each pipeline references it with a version tag. When you update the template, all pipelines using it pick up the change on next run. This is actually better than Jenkins shared libraries because templates are versioned, have their own approval process, and are visible in the UI rather than hidden in a separate repository.

❓Explain canary deployment end-to-end in a banking context.▼
A canary deployment gradually shifts traffic from old to new version. For payments-service we deploy v1.1.0 alongside v1.0.0. Harness routes 10% of transactions to the new version, monitoring error rates and p99 latency. If metrics stay healthy, Harness promotes to 25%, 50%, then 100%. If checks fail, automatic rollback fires and only 10% of users were exposed. This is why banks use canary for payment changes — even if broken, blast radius is a fraction of customers, not everyone.
❓When would canary NOT be the right strategy?▼
Canary is wrong when mixed versions cause data corruption — e.g. a DB schema migration where new code writes a field old code doesn't understand. Also wrong for breaking API contract changes. In these cases blue-green is safer — zero real users see new version until full validation and switch. Feature flags are better when the change is behavioural and can be toggled per user without redeployment.

❓Explain blue-green and when to use it over canary.▼
Blue-green maintains two complete identical environments. Blue is current production at 100% traffic. Green is deployed with new version at zero traffic. After full smoke testing of Green, traffic switches instantly via Kubernetes Service selector update completing in under 3 seconds. Blue stays alive 30 minutes for rollback. Use over canary when instant rollback is required for compliance — our accounts-service changes must be reversible within 60 seconds per FCA regulation. Also use when mixed versions cause data inconsistency.
❓How does Harness implement blue-green in Kubernetes technically?▼
Harness creates two Deployments labelled harness.io/color: blue and harness.io/color: green. Two Services: primary pointing to live colour, stage pointing to non-live for smoke testing. Traffic switch = kubectl patch service accounts-service updating selector from blue to green. Propagates to kube-proxy in 1-3 seconds with zero dropped requests. Rollback = same operation in reverse.
❓What is the main cost disadvantage of blue-green?▼
Requires double compute during deployment — both Blue and Green run full replica counts simultaneously. accounts-service with 3 replicas means 6 pods during switch window. This is why blue-green is reserved for compliance-critical services where safety justifies cost, while stateless services use rolling deployments with zero overhead.

❓Complete Interview Q&A — All 10 Days
❓What is Harness and how does it differ from GitHub Actions?▼
Harness is an enterprise CD platform built specifically for production-grade delivery. GitHub Actions is excellent for CI but hits limits for enterprise CD. Harness adds built-in canary and blue-green strategies with automatic rollback, OPA governance on every pipeline, SLO tracking with error budgets, cloud cost management, and AI assistance (AIDA). The key difference: Harness treats the deployment itself as a first-class concern — it knows what healthy looks like, monitors it, and rolls back automatically if health checks fail.
❓When would you use canary vs blue-green vs rolling?▼
Canary is best for high-risk changes to critical services like payments — gradually expose a small percentage of real users, monitor closely, commit when confident. Blue-green is best when you need instant rollback capability — deploy a complete new environment and switch traffic instantly, keep old environment alive for quick reversion. Rolling is the default for stateless services where brief mixed-version periods are tolerable. In our bank platform: canary for payments-service (financial risk), blue-green for accounts-service (compliance requires instant rollback), rolling for fraud and notification services.
❓What is an OPA policy in Harness and how does it work?▼
OPA (Open Policy Agent) is a policy-as-code engine using the Rego language. Harness evaluates every pipeline step against configured OPA policies before execution. We used it to enforce: production requiring two approvers from different teams, no self-approval of your own code, Checkov HIGH findings blocking any deployment, no production deployments outside business hours, and mandatory resource limits on all K8s manifests. All policies are stored in Git giving a complete audit trail of every governance decision — which is exactly what compliance teams require for SOX, PCI-DSS, and other regulations.
❓What is the difference between SLI, SLO, and SLA?▼
An SLI (Service Level Indicator) is the actual metric measured — for example the fraction of HTTP requests returning 200 status. An SLO (Service Level Objective) is the internal target — 99.9% of requests must succeed. An SLA (Service Level Agreement) is the external contract with customers carrying financial penalties if breached. In Harness SRM we configure SLIs and SLOs, which track error budgets (the amount of failure you can afford). When 90% of the budget is consumed, deployments are automatically blocked before any SLA violation can occur.
❓How would you approach migrating from Jenkins to Harness in 90 days?▼
Three phases. Phase 1 (Days 1-30): Install Delegates in non-prod, migrate one non-critical service as POC, run Jenkins and Harness in parallel, train all teams. Phase 2 (Days 31-60): New services go Harness-only from day one, migrate existing services in batches starting with lowest risk, keep each Jenkins pipeline as fallback for 2 weeks before removing. Phase 3 (Days 61-90): Migrate critical services like payments with extended parallel running and validated rollback testing, then decommission Jenkins. Key principle: never leave teams without a rollback option during the transition window.
❓What are DORA metrics and how does Harness improve them?▼
DORA metrics are the four industry-standard measures of DevOps performance. Deployment Frequency — how often you deploy. Lead Time for Changes — code commit to production. MTTR — time to recover from incidents. Change Failure Rate — percentage of deployments causing incidents. Harness improves all four: automated pipelines increase deployment frequency, removing manual steps cuts lead time, automatic rollback drastically reduces MTTR, and canary deployments catch issues before full rollout reducing change failure rate. Measuring these before and after migration provides concrete ROI justification for leadership.
