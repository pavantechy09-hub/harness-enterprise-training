🧠 The Canary Pattern Explained
⚠️ The name comes from "canary in a coal mine" — miners sent a canary first to detect toxic gas. You send a small percentage of traffic first to detect problems before full deployment.
Canary Deployment Flow — payments-service v1.1.0
Deploy v1.1.0
alongside v1.0.0
→
10% traffic
→ v1.1.0
→
Monitor 5min
errors / p99
→
25% → 50%
→ 100%
⟵ If error rate spikes at any point → automatic rollback to v1.0.0 → only affected users = the canary %
✅ When to Use Canary
• High-risk changes to payment or auth services
• New fraud detection algorithms
• Any change affecting money movement
• When you need real production validation
• When synthetic tests aren't sufficient
❌ When NOT to Use Canary
• Database schema migrations (affects all pods)
• Breaking API contract changes
• Infrastructure changes (use blue-green)
• When mixed versions cause data corruption
• Config-only changes (use rolling)
🎯 Auto-Rollback Triggers
• Error rate > 1% (normal: 0.01%)
• p99 latency > 5 seconds
• HTTP 5xx rate spike
• Custom metric threshold breach
• Health check endpoint returning non-200
🏦 Bank Use Case — Real Scenario
Scenario: payments-service v1.1.0 includes new fraud pre-check before payment authorisation. The change is complex — if it has a bug, it could block legitimate payments.

Without canary: Deploy to all 3 pods simultaneously. If bug exists, 100% of payment attempts fail. All customers affected. Emergency rollback takes 8 minutes. £2.3M in blocked transactions.

With canary: 10% traffic to v1.1.0. Bug detected after 47 seconds — error rate 23% on new pod. Harness automatically rolls back. Only 10% of traffic was affected. No manual intervention. Total impact: 47 seconds × 10% of users = minimal.

This is why enterprise banks use canary for every payment service change.

pipeline:
  name: "Day 5 - Canary Deployment (payments-service)"
  identifier: day5_canary_payments
  projectIdentifier: bankplatform
  orgIdentifier: default

  stages:
    # ─── STAGE 1: Deploy Canary (10%) ───────────────────────
    - stage:
        name: "Canary — 10% Traffic"
        identifier: canary_10
        type: Deployment
        spec:
          deploymentType: Kubernetes
          service:
            serviceRef: payments_service
            serviceInputs:
              serviceDefinition:
                type: Kubernetes
                spec:
                  artifacts:
                    primary:
                      primaryArtifactRef: primary
                      sources:
                        - identifier: primary
                          spec:
                            tag: <+pipeline.variables.imageTag>
          environment:
            environmentRef: dev
            infrastructureDefinitions:
              - identifier: dd_stg_k8s
          execution:
            steps:
              - step:
                  name: Canary Deployment
                  identifier: canary_deployment
                  type: K8sCanaryDeploy
                  spec:
                    instanceSelection:
                      type: Count
                      spec:
                        count: 1  # Deploy 1 pod out of 3 = 33% (Harness handles % calc)
                    skipDryRun: false
            rollbackSteps:
              - step:
                  name: Canary Rollback
                  identifier: canary_rollback
                  type: K8sCanaryDelete
                  spec: {}

    # ─── STAGE 2: Verify Canary Health ──────────────────────
    - stage:
        name: "Verify Canary Health"
        identifier: verify_canary
        type: CI
        spec:
          execution:
            steps:
              - step:
                  name: Check Error Rate
                  type: Run
                  spec:
                    image: curlimages/curl:latest
                    command: |
                      echo "Waiting 30s for canary traffic..."
                      sleep 30

                      # Check health endpoint on canary pod
                      HEALTH=$(curl -s -o /dev/null -w "%{http_code}" \
                        http://payments-service.payments.svc.cluster.local/health)

                      if [ "$HEALTH" != "200" ]; then
                        echo "CANARY HEALTH CHECK FAILED: $HEALTH"
                        exit 1
                      fi

                      echo "Canary health check PASSED: $HEALTH"

              - step:
                  name: Check Metrics (Simulated)
                  type: Run
                  spec:
                    image: alpine:3.18
                    command: |
                      echo "Error rate: 0.3% (threshold: 1%) ✓"
                      echo "p99 latency: 280ms (threshold: 5000ms) ✓"
                      echo "Canary is healthy — ready to promote"

    # ─── STAGE 3: Promote to Full ───────────────────────────
    - stage:
        name: "Promote — Full Rollout"
        identifier: promote_full
        type: Deployment
        spec:
          deploymentType: Kubernetes
          execution:
            steps:
              - step:
                  name: Canary Delete (cleanup canary pods)
                  type: K8sCanaryDelete
                  spec: {}
              - step:
                  name: Rolling Deploy (full rollout)
                  type: K8sRollingDeploy
                  spec:
                    skipDryRun: false

  variables:
    - name: imageTag
      type: String
      default: "latest"

  notificationRules:
    - name: Canary Alerts
      pipelineEvents:
        - type: StageFailed
        - type: PipelineSuccess
      notificationMethod:
        type: Slack
        spec:
          webhookUrl: <+secrets.getValue("slack_webhook_url")>
          message: |
            Canary deployment <+pipeline.status>
            Service: payments-service
            Tag: <+pipeline.variables.imageTag>
            Stage: <+stage.name>

❓Explain canary deployment end-to-end in a banking context.▼
A canary deployment gradually shifts traffic from the old version to the new version. In our bank, we deploy payments-service v1.1.0 alongside the existing v1.0.0. Harness first routes 10% of payment transactions to the new version, monitoring error rates and p99 latency in real time. If metrics stay within thresholds — error rate below 1%, p99 below 5 seconds — Harness automatically promotes to 25%, then 50%, then 100%. If any health check fails, Harness triggers automatic rollback and only 10% of users were exposed to the bad version. This is why enterprise banks use canary for every payment service change — even if the new version is broken, the blast radius is a fraction of customers rather than everyone.
❓How does Harness implement canary in Kubernetes technically?▼
Harness creates a second Deployment object with a -canary suffix, running the new image version with a separate pod selector label harness.io/track: canary. The stable Deployment continues running with harness.io/track: stable. The Kubernetes Service selector matches both track labels, so requests naturally distribute between old and canary pods based on the ratio of running pods. When the canary is 1 pod out of 3 total, approximately 33% of traffic goes to canary. After successful promotion, Harness deletes the canary Deployment and performs a rolling update on the stable Deployment to the new version.
❓When would canary NOT be the right strategy?▼
Canary is inappropriate when mixed versions would cause data corruption — for example, a database schema migration where new code writes a field that old code doesn't understand. It's also inappropriate for breaking API contract changes where old clients would fail on new responses. In these cases, blue-green deployment is safer because you can fully validate the new version with zero production traffic before any cutover. Feature flags are sometimes better than canary when the change is purely behavioural and can be toggled per user without redeployment.
❓What metrics do you monitor during a canary window?▼
The primary metrics are error rate (HTTP 5xx responses as a percentage of total requests), p99 latency (99th percentile response time — catches slow outliers that average hides), and health check status (is the /health endpoint returning 200). In a bank we would also monitor payment success rate (are transactions actually completing), fraud check pass rate (no unexpected spikes), and database connection errors. The canary window duration matters — too short and you miss edge cases, too long and you delay the rollout. Typical banking teams use 5-15 minutes per stage.
Blue-Green Deployment

🔵🟢 Instant Traffic Switch

✅ Why Blue-Green for accounts-service
Account balance queries are compliance-critical. Any incorrect balance shown to a customer triggers regulatory reporting. Blue-green lets us fully validate v1.2.0 with zero production traffic before any customer sees it, and provides instant 2-second rollback if issues appear after the traffic switch.
💡 Key Difference from Canary
Canary: real users gradually shifted to new version. Real traffic validates the deployment. Some users see issues during observation window.

Blue-Green: full validation with zero real users before switch. Instant switch means either 0% or 100% on new version. No gradual exposure.
💰 Cost Consideration
Blue-Green requires double the infrastructure during the deployment window — both Blue and Green running simultaneously. For accounts-service: 3 Blue pods + 3 Green pods = 6 pods total during switch. After 30 minutes, Blue is decommissioned and cost returns to normal.
🏦 Bank Use Case — Compliance Requirement
Regulatory requirement: Any change to account balance calculation logic must be fully reversible within 60 seconds if a defect is detected post-deployment. This is a PCI-DSS and FCA (UK Financial Conduct Authority) requirement.

How blue-green satisfies this:
1. Green (new version) deployed and smoke-tested in isolation
2. Traffic switched to Green in 2 seconds (Service selector update)
3. Blue kept running for 30 minutes at zero traffic
4. If compliance team flags an issue: traffic switched back to Blue in 2 seconds
5. Blue version still running, data intact, customers see correct balances

Result: Deployment is provably reversible within the 60-second regulatory requirement. Audit trail shows who approved the switch and when. Compliance team satisfied.

                      import httpx

                      assert r.status_code == 201, f'Create failed: {r.status_code}'

⚡ The Traffic Switch — How Fast Is It?
Harness runs kubectl patch service accounts-service -p '{"spec":{"selector":{"harness.io/color":"green"}}}'. Kubernetes propagates this change to kube-proxy on all nodes within 1-3 seconds. New requests immediately go to Green pods. In-flight requests to Blue complete normally. Zero dropped requests. Zero downtime. This is why blue-green is preferred for compliance-critical services over canary.
❓Explain blue-green deployment and when you would use it over canary.▼
Blue-green maintains two complete, identical environments. Blue is the current production environment serving 100% of traffic. Green is deployed with the new version but receives zero production traffic initially. After full smoke testing and validation of Green in isolation, traffic is switched instantly — a Kubernetes Service selector update that completes in under 3 seconds. Blue remains alive for 30 minutes as an instant rollback option. I use blue-green over canary when instant rollback is required for compliance — in our bank, accounts-service changes must be fully reversible within 60 seconds per FCA regulation. I also use it when mixed versions would cause data inconsistency, making canary's gradual shift dangerous.
❓How does Harness implement blue-green in Kubernetes technically?▼
Harness creates two Deployments — one labelled harness.io/color: blue and one harness.io/color: green. It maintains two Services: a primary Service whose selector points to whichever colour is live, and a stage Service that always points to the non-live colour for smoke testing. The traffic switch is a kubectl patch on the primary Service selector, updating it from blue to green. This propagates to kube-proxy on all nodes in 1-3 seconds with zero dropped requests. Rollback is the same operation in reverse — patch the selector back to blue. The stage Service remains unchanged throughout, allowing smoke tests on the non-live version at any time.
❓What is the main cost disadvantage of blue-green?▼
Blue-green requires double the compute resources during the deployment window — both the Blue and Green Deployments run their full replica counts simultaneously. For accounts-service with 3 replicas, this means 6 pods during the switch window instead of the normal 3. In a large enterprise with 200 services, if they all used blue-green simultaneously the cost would be significant. This is why blue-green is reserved for compliance-critical services where the safety guarantee justifies the cost, while stateless low-risk services use rolling deployments that have zero extra cost overhead.
❓How do you handle database migrations with blue-green deployments?▼
Database migrations are the hardest part of blue-green. The safest pattern is expand-contract: first deploy a migration that adds new columns without removing old ones — now both Blue and Green work. Then switch traffic to Green. Then deploy a second migration that removes the old columns once Blue is decommissioned. This requires two deployments but ensures zero downtime and instant rollback at every stage. Harness can orchestrate this as a multi-stage pipeline where the schema migration is a separate stage before the application deployment stage. We configure a specific step to run database migrations as a Kubernetes Job before the Deployment rollout begins.
Rolling Deployment

🔄 Pod-by-Pod Replacement

Replace pods one at a time. Zero downtime. Zero extra cost. Default Kubernetes strategy. Best for stateless services where brief mixed-version windows are acceptable.

✅ Used for fraud-service — stateless ML scoring, low risk, no compliance requirements for instant rollback.
🔄 How Rolling Works
Kubernetes replaces pods one by one (or in configurable batches).

3 pods running v1.0.0:
[v1.0.0] [v1.0.0] [v1.0.0]

Replace pod 1, wait for Ready:
[v1.1.0] [v1.0.0] [v1.0.0]

Replace pod 2, wait for Ready:
[v1.1.0] [v1.1.0] [v1.0.0]

Replace pod 3:
[v1.1.0] [v1.1.0] [v1.1.0]

Brief period with mixed versions is acceptable for fraud-service.
⚙️ Key Config Parameters
maxSurge: 1 — create 1 extra pod before killing old ones. Ensures capacity never drops.

maxUnavailable: 0 — never have fewer than desired count running. Zero downtime guaranteed.

readinessProbe — pod must pass health check before traffic is routed to it. Prevents unhealthy pods from receiving requests.
✅ When to Use Rolling
• Stateless services (fraud scoring, notifications)
• Low-risk changes (bug fixes, minor updates)
• When mixed versions are safe (no shared state)
• When you want zero extra infrastructure cost
• Config or environment variable changes
src/fraud-service/helm/templates/deployment.yaml

Feature Flags

🚩 Deploy Code Without Activating It

Ship code to production but control who sees it. Activate for 5% of users. Monitor. Increase gradually. Kill switch available instantly — no deployment needed to roll back.

✅ Used for notification-service — new SMS provider deployed to ALL pods but only enabled for 5% of users via feature flag.
🚩 What Feature Flags Solve
Traditional deployments activate code for everyone or no one. Feature flags decouple deployment from activation.

New SMS provider is deployed to production. Flag controls who uses it:
• 5% of users → Twilio (new)
• 95% of users → Legacy SMS

If Twilio fails: flip flag OFF instantly. No deployment, no rollback, no 3am incident.
💻 Code Pattern
if ff.isEnabled("new-sms-provider", userId):
    send_via_twilio(message)
else:
    send_via_legacy(message)

The flag evaluation happens at runtime. No code change needed to enable or disable. The kill switch is literally a checkbox in Harness FF UI.
🎯 Gradual Rollout
Rollout percentage controlled in Harness UI:

Day 1: 5% of users (monitoring)
Day 2: 25% if metrics good
Day 3: 50%
Day 4: 100% → flag removed from code

Each stage: watch delivery rate, latency, errors. Metrics gate the progression automatically.

🏦 Decision Framework for Enterprise Banking
Use Canary when: Payment logic, auth service, fraud algorithm changes. You need real user validation but can tolerate 10% exposure window.

Use Blue-Green when: Compliance requires instant rollback. Account balance calculations. Any service with regulatory audit requirements.

Use Rolling when: Stateless services, notification systems, internal tools. Zero extra cost justified for low-risk changes.

Use Feature Flags when: A/B testing, gradual feature rollout, kill switch required without redeployment. Best for UX changes and new integrations.
❓ Interview Q&A — All Deployment Strategies
❓What is the key difference between canary and blue-green deployments?▼
The fundamental difference is how real users are exposed to the new version. Canary gradually shifts real production traffic — 10% of actual customers use the new version first. If it breaks, 10% of customers are affected before automatic rollback. Blue-green deploys to a completely separate environment with zero real traffic, validates it fully via smoke tests, then switches all traffic instantly. No real users are exposed to the new version until the switch, and rollback means switching back — completing in under 3 seconds. Blue-green provides stronger safety guarantees for compliance-critical services but costs double infrastructure during the deployment window.
❓A deployment is running on 3 pods. You roll out a new version and notice the error rate spiking. Walk me through what happens with each strategy.▼
Rolling: 1 pod is already on v2 and serving errors. The rolling update pauses (if configured). You must manually roll back. During this time, 33% of traffic is on broken v2. Manual recovery takes 5-10 minutes.

Canary: The canary pod (1 pod) detects the error spike. Harness automatically rolls back the canary pod. Only the canary traffic percentage (33%) was affected, and rollback completes in seconds. Users on stable v1 were never impacted.

Blue-Green: The switch already happened — all 3 pods are on v2 (Green). You detect the issue and instantly switch the Service selector back to Blue (which was kept alive). 2-3 second rollback. Zero debugging required — just run the switch-back command.

The interview insight: Which is "best" depends on what you optimise for — blast radius (canary), rollback speed (blue-green), or infrastructure cost (rolling).
❓How do feature flags differ from deployment strategies? When would you choose one over the other?▼
Deployment strategies control which version of code is running — all users get the same code, but gradually or all-at-once. Feature flags control which features are active for which users — all users run the same deployed code, but the flag determines which code path executes.

Choose feature flags when: you want per-user or per-segment targeting (premium customers get feature first), you need a kill switch without redeployment, you're doing A/B testing, or you're integrating a new third-party service where you want controlled rollout.

Choose deployment strategies when: the change is infrastructure-level (can't be wrapped in a flag), the change affects all code paths equally, or you want real traffic to validate performance characteristics of the new version. In practice, enterprises use both together — deploy with canary, then enable features with flags.
❓How does Harness automatically decide when to roll back a canary?▼
Harness uses Continuous Verification connected to monitoring tools like Prometheus, Datadog, or Dynatrace. During the canary observation window, it compares metrics from the canary pods against the stable pods as a baseline. If the canary's error rate is statistically significantly higher, or if p99 latency exceeds the configured threshold, Harness triggers automatic rollback without human intervention. The thresholds are configured per service — we set error rate threshold at 1% (normal is 0.01%) and p99 threshold at 5 seconds for payments-service. Harness also supports custom metrics — you can set a rollback trigger on business metrics like payment success rate, not just technical metrics like error rate.
❓What are DORA metrics and how do deployment strategies affect them?▼
DORA metrics measure DevOps performance. Deployment frequency increases because canary and feature flags reduce the risk of each deployment — smaller, safer deployments can happen more often. Lead time for changes decreases because automated canary analysis removes the manual observation period. MTTR drops dramatically — automatic canary rollback takes seconds versus 45 minutes of manual investigation. Change failure rate decreases because canary catches issues before full rollout and blue-green enables instant recovery. In our bank platform migration from Jenkins to Harness with proper deployment strategies, we target: deployment frequency 10x increase, lead time 75% reduction, MTTR 80% reduction, change failure rate from 15% to under 5%.
